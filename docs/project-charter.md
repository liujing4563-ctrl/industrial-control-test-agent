# Project Charter

## 项目背景

本项目面向嵌入式控制板与 PLC I/O 测试场景，解决现场测试工程师在硬件验证、故障定位和回归验证中的重复性高、过程不可追踪、跨领域知识难沉淀的问题。

## 目标用户

- 嵌入式测试工程师
- PLC 验证工程师
- 自动化测试负责人
- 系统集成工程师

## 真实工业痛点

- MCU 与 PLC I/O 领域测试经验分散、难以复用。
- 设备操作与故障分析高度依赖人工经验。
- 多阶段测试缺少统一案件状态与证据链。
- 设备修改与回归验证缺乏审计与审批机制。

## 产品目标

构建一个可扩展、可验证的多智能体测试工程平台，支持 MCU 与 PLC 两种领域能力包，以统一案件状态驱动测试设计、故障诊断、自主补测和回归验证。

## 比赛 MVP

- 支持 MCU 嵌入式控制板测试。
- 支持 PLC I/O 测试。
- 提供统一的案件生命周期与共享 CaseState。
- 提供 MCU 专业 Agent、PLC 专业 Agent、Diagnosis/Critic Agent。
- 提供测试能力包与 MCP 能力接口。
- 提供确定性 Policy Engine 与 Test Runner。
- 提供 Evidence Store 及人工审批节点。
- 提供故障诊断与补测闭环。

## 长期愿景

成为工业测试团队的标准工程平台，将测试 SOP、故障经验、判定规则与回归要求沉淀为可执行能力包，支持多种设备、工具和模型供应商，并提供完整审计链。

## 非目标

- 不支持任意工业设备。
- 不接入真实生产线控制。
- 不自动修改 PLC 程序或真实固件。
- 不执行用户上传的任意 Python/Shell 代码。
- 不建设公共 Skill 市场。
- 不实现完整企业权限系统。
- 不依赖多 Agent 自由对话作为核心逻辑。

## 成功判据

- 形成清晰的架构和系统设计文档。
- 形成统一的 Schema 与领域模型契约。
- 形成核心 Python 接口和抽象骨架。
- 形成三位成员明确的工作拆分与边界。
- 所有骨架代码可以被导入，架构测试通过。

## 数据合规边界

- 只保存测试过程必要的状态和证据引用。
- 原始设备数据视为只读 Evidence，不允许 Agent 修改。
- 不在本阶段实现真实设备或者外部数据存储集成。

## 三人能力组合

- 系统负责人：顶层架构、案例工作流、Agent 平台、文档与集成。
- MCU 专家：MCU 测试能力包、MCU Agent、MCU 模拟器、MCU 领域模型。
- PLC 专家：PLC I/O 能力包、PLC Agent、PLC 仿真、PLC 领域模型。
