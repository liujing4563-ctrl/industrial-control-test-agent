# 架构设计

## 六层架构

1. 测试工程工作台（Test Engineering Workspace）
   - 提供测试案件视图、审批界面和结果查看。
   - 不直接执行设备操作。

2. 案件与工作流编排器（Case & Workflow Orchestrator）
   - 维护统一 CaseState。
   - 控制案件阶段、审批中断与回归闭环。
   - 负责工作流调度和状态持久化。

3. 多智能体推理层（Multi-Agent Reasoning Layer）
   - 包含执行智能体、专业智能体、诊断与审查智能体。
   - 仅生成结构化 ActionIntent 和推理结果。
   - 不直接调用设备工具。

4. 策略引擎与测试执行器（Policy Engine & Test Runner）
   - 策略引擎校验 ActionIntent、权限与风险。
   - 测试执行器执行批准后的动作。
   - 所有写操作默认进入审批。

5. 技能注册中心与 MCP 网关（Skill Registry & MCP Gateway）
   - 技能注册与版本管理。
   - MCP Gateway 提供工具发现、输入输出校验、白名单和限流。

6. 证据与治理平面（Evidence & Governance Plane）
   - 保存原始 Observation 与 Evidence。
   - 记录审批、变更提案、回归结果。
   - 提供审计追踪与后续分析数据。

## 层间依赖

- 上层只能依赖下层，不反向。
- 智能体只能依赖领域模型、技能栈和运行时抽象。
- 策略、执行器、MCP 网关不能依赖智能体推理实现。

## 数据流

1. 智能体读取 CaseState，生成 ActionIntent。
2. 编排器接收 ActionIntent，进入策略校验。
3. 策略通过后交给执行器。
4. 执行器调用 MCP 网关，执行测试服务。
5. 测试服务返回 Observation，写入证据存储。
6. 编排器更新 CaseState、假设和回归状态。

## 控制流

- 智能体发起请求，编排器判断是否继续、中断或审批。
- 所有写操作进入审批节点；人工批准后恢复执行。
- 当 Evidence 足够或达到终止条件时，案件结束。

## 失败处理

- 任何节点失败后记录失败事件。
- 如果执行器或设备执行失败，进入重规划或升级处理。
- 如果审批拒绝，案件进入人工中断状态。

## 人工审批

- 修改真实设备前必须生成变更提案（ChangeProposal）。
- 审批通过后，由执行器通过 MCP 网关调用。
- 审批结果写回 CaseState 与 Evidence。

## 回归闭环

- 变更执行后自动触发回归测试。
- 回归结果作为证据记录并指向原始变更提案。
- 回归通过后关闭案件。
